Sesion del 2026-04-09

- Se consolidó el flujo de múltiples propietarios por unidad:
  - propietario principal
  - copropietarios
  - apoderados
  - voto
  - convocatorias
- Se implementó el flujo de poder para apoderados:
  - se puede guardar el apoderado sin poder cargado
  - mientras no esté aprobado no puede votar ni recibir convocatoria
  - se genera link público para cargar el poder
  - el poder puede enviarlo propietario, copropietario o el mismo apoderado
  - el administrador lo revisa desde dashboard
  - si lo aprueba se habilitan voto y convocatoria
  - si lo rechaza quedan razones visibles para reenviar
- Se corrigieron varios problemas operativos del entorno:
  - procesos viejos de Vite/API
  - archivos `.js` mezclados con `.tsx`
  - páginas en blanco por módulos viejos en memoria
- Se avanzó fuerte en el módulo de preparación de asamblea:
  - configuración formal de la asamblea persistente
  - orden del día persistente
  - anexos y documentos persistentes con visor
  - convocatoria con destinatarios y trazabilidad
  - acceso e identidad con matriz por unidad
  - revisión final con estado `riesgo alto`, `en preparación`, `lista para convocar`, `lista para abrir`
- Se cargó una base hipotética amplia en `apps/api/data/app-db.json` para pruebas:
  - 100 unidades
  - 147 personas relacionadas
  - apoderados aprobados y pendientes
  - convocatorias y accesos en varios estados
  - backup en `apps/api/data/app-db.before-hypothetical-seed.json`
- Se hizo una auditoría UI/UX de la plataforma y el usuario estuvo de acuerdo con los hallazgos:
  - demasiadas tarjetas con el mismo peso
  - dashboard tipo collage
  - poca jerarquía
  - demasiada verticalidad
  - estados sin acción clara
- Se empezó una refactorización visual real:
  - dashboard más ejecutivo
  - lista de unidades con salud operativa
  - ficha de unidad con acordeones y resumen operativo
  - cabina de preparación con mejor progresión y rail lateral
- Se propuso branding del producto:
  - nombre provisional aceptado: `Quorum`
  - se creó un `BrandSignature` con isotipo simple y wordmark
- El usuario compartió referencias visuales más limpias tipo dashboard SaaS moderno y pidió movernos hacia esa dirección.
- Se intentó varias pasadas del dashboard:
  - hero superior mejorado
  - alertas accionables
  - sidebar izquierda
  - topbar más ligera
  - composición más tipo plataforma
- Aun así, el usuario considera que el dashboard sigue viéndose desordenado y “regado”.
- La última dirección acordada fue tomar como referencia un dashboard estilo SaaS limpio con:
  - sidebar clara
  - topbar ligera
  - canvas central respirado
  - menos bloques dispersos
  - jerarquía más fuerte de navegación y acciones
- El usuario luego pidió revertir los últimos cambios de estilo porque no le gustaron.
- Se revirtieron las capas visuales recientes en las pantallas más visibles:
  - dashboard
  - landing
  - login
  - registro
  - sala
- No se revirtió funcionalidad, solo la dirección visual reciente.
- Quedó claro que mañana no conviene seguir “parchando” estilos:
  - primero definir el diagrama/layout exacto del dashboard
  - luego implementar sobre esa base
- El usuario quiere que mañana retomemos sabiendo que el próximo paso es diseñar correctamente la estructura, no seguir iterando a ciegas en tarjetas.
